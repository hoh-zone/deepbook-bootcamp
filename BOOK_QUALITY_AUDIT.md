# DeepBook 书稿质量审查记录

审查日期：2026-05-06

本文件记录当前书稿的质量状态、已修订项和剩余风险。它不是读者正文，而是后续继续打磨书稿时的编辑基线。

## 当前结论

书稿已经具备完整 mdBook 结构：`ch00` 到 `ch16`、SUMMARY 导航、官方文档基线、源码定义地图、学习路线、代码目录和 GitHub Pages 构建配置。`mdbook build` 当前通过。

但“能构建”不等于“能出版”。当前质量已经从纯大纲推进到可读书稿，仍需要继续补强三类内容：可运行代码、索引附录、后段应用章节的真实实现细节。

## 本轮审计发现

1. **ch01-ch03 模板句影响阅读感**

   审计时发现 `这一节只保留必要入口` 和 `读源码时先确认对象、函数签名和事件名称` 在早期章节大量重复。它们能导航，但会让书稿显得像批量生成的文档。

2. **SDK 章安装配置不够落地**

   `ch12-02` 原文只给依赖方向，没有安装命令、最小验证脚本、sandbox 配置和私钥边界。对“SDK 集成专题”来说，这会让读者无法判断自己是否真正接入成功。

3. **ch15 部分小节仍像 TODO**

   `全书项目 README`、`代码运行手册`、`附录` 原本在描述“应该包含什么”，不像正式章节。它们需要改成交付标准、推荐结构、验收方式和可查证证据。

4. **代码目录的运行口径不足**

   ch14/ch15 代码目录原来多为模块清单，例如交易终端、做市机器人、部署 compose、E2E 测试等只有几行说明。读者无法知道这些目录是可运行示例、实现蓝图，还是待补齐模板。

5. **质量记录本身失真**

   旧审计文件写过“已批量移除 ch01-ch03 早期模板句”，但实际仍有残留。本轮已把质量记录改为可验证状态，避免把未完成事项写成完成。

## 出版级判定标准

一个小节可以进入出版级状态，至少要满足：

- 有明确读者场景或真实工程问题。
- 有官方文档、源码、命令或测试作为事实依据。
- 关键结构体、函数、事件或 handler 不只给链接，必要时直接摘入正文解释。
- 能说明对象、资金、事件或读模型如何变化。
- 写清应用侧如何用：SDK、PTB、dry run、Indexer、UI、监控或部署。
- 明确失败路径和风险边界，尤其是 Margin、Predict、Flash Loan、Indexer lag 和本地 sandbox。
- 代码目录明确标注状态：可运行、模板、蓝图、待补齐，不把蓝图伪装成成品。
